User → Command → Subagent → Skill の形で機能を呼び出すのが基本。

## Mode

- default: 読み書きや実行に毎回確認を求める
- accept edits on: 読み書きのみを承認なしで進める
- plan mode: 読み取り専用
- auto mode: コマンド実行などの全ての動作を承認なしで進める

最初はデフォルトで使って、セキュリティの設定が作れたら確認を省略してもいい。

明確な作業の指示以外はPlanモードから入るといい。
どっちみちまずは指示を明確にしないといけない。

小さなタスクでも少しでも曖昧な部分があるならPlanモードから始める。

MVPを作るのは大きなタスクなので、Planモードを使った方がいい。

一連の作業が終わったらPlanかDefaultに戻るのが基本。
そうしないと話している最中に勝手に編集し始めたりする。

## Rule

`CLAUDE.md` または `rules/*.md` に記載したプロンプトはすべてのプロンプトの前提として読み込まれる。
Claude Codeは会話を終えるとメモリリセットされるので、常に覚えておいて欲しいことを書く。

全てのタスクに一貫した振る舞いを適用したいならルールを設定する。
それぞれのタスクのコンテキストについてはスキルやサブエージェントとして定義する。

巨大なルールはコンテキストウィンドウを圧迫しすぎる。
最大でも100行程度に抑えて、Claudeが必要に応じて調べにいけるドキュメントやスキルを整備する。

https://qiita.com/nogataka/items/0cd0851556572b4758ba

「制約」が重要。

`CLAUDE.md`などのルールが大きくなるとそれだけでコンテキストウィンドウを圧迫してしまうので、全部で100行程度に抑えたい。
代わりに`docs/`などに設計情報などを整備して、「必要に応じて`@docs`フォルダの書類を確認する」などと指示しておけば読み込んでくれる。ただし、確実に取り込みたいルールはルールに書いた方がいい。コーディング規約や操作の制限など。

ただ、情報自体は一旦文脈として渡しておいて、パフォーマンスが落ちたりコストがかさんだりするようなら抑えるくらいでいい。まずは文脈を作るのが優先。

`@path/to/import`がインポート構文になっている。

効果的なルールの書き方は公式ドキュメントを参照する。
https://code.claude.com/docs/ja/best-practices#効果的な-claude-md-を書く

> 各行について、次のように尋ねます。「これを削除すると Claude が間違いを犯しますか？」 そうでない場合は、削除します。

設定ファイルや型ファイルなどの共有部分については、変更するルールを明確にしておくと競合が起きにくい。
単純なのは触らない場所を明確にして、変更が必要なら使用者がまとめて設定する。

共有部分については「共通部分を触りたい場合は担当者に委託して、そちらの作業を取り込んでから続きを進めてください」のような指示を入れるのもあり。
あるいは一旦別ファイルを作っておいて、あとからリファクタリングでもいい。コーディング自体は早くなっているので、非効率でも管理しやすい方法で進めるべき。

人間みたいに「察して」は通じない。
全てを明文化する必要がある。

path-scoped rulesがいつ適用されているかは`/memory`で読み込まれているファイル一覧を見ると確認できる。

余計なルールは削除しないと邪魔になる。

## Slash Command

ユーザーが呼び出せる拡張コマンド。
`/clear`や`/usage`のような機能を自分で作れる。

スラッシュコマンドの層は薄い方がいい。
エージェントやスキルを別途用意してそれを呼び出すか、単にスキルとして作る。

## Skill

指示の短縮形。
特定のスコープやワーフクローに制限されたルールのように機能する。

Skillとして定義して、Slash Commandで呼び出す。
ただし、Claudeが勝手に呼び出すことがあるので、常に明示的に呼び出したいならスキルにはせずにコマンドだけ作るといい。
まずはコマンドを作って、Claude側でうまいこと使って欲しいならスキルにする、という段階的な発展がおすすめ。

https://code.claude.com/docs/ja/skills

まずは公式のSkillを使えばいい。Pluginとして導入できる。
https://github.com/anthropics/skills

明示的に呼ばなくても、文脈として持っていて欲しいものをスキルにする。コマンドというよりもルールに近い。
実行能力を補う。Claudeがどういう作業をするときに、どういう手順で進めるべきか、を考える。「手順」が重要。

## Hook

特定のイベントで発火する処理。

- ツール実行前
- ツール実行後
- メッセージ送信時
- Claudeの応答終了時
- コンテキスト圧縮前
- 権限リクエスト

ルールと違って、`settings.json`に書いて必ず実行される。

## Subagent

メインのClaudeとはコンテキストの異なる指示を実行する。
メインとは別に専門家がいるイメージ。

コンテキストが重い処理をメイン処理から切り離すのに適している。

- コンテキストを保持する
- ツールを制限する
- 設定を再利用する
- 動作を特化させる
- コストを制御する（メインと違うモデルでコストを抑えるとか）

サブエージェントはプロジェクトを跨いで作る。

https://code.claude.com/docs/ja/agents

メインラインとは異なるルールを書く。
コンテキストを「あえて」共有しない。第三者機関がサブエージェントと考えるべきかも。プロジェクトの都合だけで動かずに、法律とか、別のコンテキストで判断して正論を言う。

テストとかはコンテキスト自体が同じなのでスキルでいい。単にforkして実行するだけで、サブエージェントでなくていい。
ただ、QAレベルの品質テストとかはあえてコンテキストを共有しないサブエージェントにする手もある。

ペルソナの立場で意見を集めたい時に、それぞれのコンテキストで好き勝手言わせるのもいい。

エージェントが必要になるのは、コンテキストを分離するとき。
役割があるからといってとりあえずエージェントにするのは違う。

メインの役割（ソフトウェア開発なら開発者）と切り離したコンテキストで動作させる。
よく使われるのは、レビューやテスト。第三者視点で動かした方がいい。監査のような役割。

## Agent Teams

サブエージェントを並行処理して連携させる機能。

容量をバカ喰いするので注意。

Subagentの欠点を改善する目的がある。
まだ実験的な機能。

- 順次処理になるので時間がかかる
- 結果を返すが、過程がわからない
- 割り込みができない
- Subagent同士は連携しない

## Plugin

Skill, Slash Command, Hook, などを簡単にインストールできるパッケージにする。

出かけてる間に時間決めて、2時間でできるとこまでやっといて、みたいな指示ってできるのかな？

デスクトップのファイルをスマホから操作することもできるみたい。/remote-controlで。

プラグインは何を入れるかよりも、何を入れないかがまず重要。
特にフック入りのプラグインは勝手に反応するので、導入するだけでセッションが重くなることもある。

公式のプラグインとコミュニティのプラグインがある。
まずは `anthropics/claude-plugins-official` の中だけで探すといい。
`anthropics/claude-plugins-community`のプラグインもプラグインを作る勉強にはなる。

- まずは公式から
- 自分の役に立つか
- フックがあるなら、常時機動コストを許容できるか
- `SKILL.md` を読んで、何をするか理解できるか
- 自分の `CLAUDE.md` と競合・重複しないか

claudeスキルを増やすと、使わないスキルでもdescriptionなどのメタデータはコンテキストに毎回読み込まれる。基本的にはプロジェクトごとに必要なものだけを取り込むべき。
スキルが多いと誤って呼び出されるリスクもある。関係ないプロジェクトでVercelやSupabaseにアクセスしにいくとか。

公開されているスキルを導入するには、
配布元をマーケットプレイスに登録してプラグイン機能で導入するか、
単にローカルの.claude/skillsにファイルを置く。
