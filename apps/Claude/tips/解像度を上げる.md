# 解像度を上げる Resolving Ambiguity

**Claudeは「指定されていない部分を平均値で埋める」**性質があります。プロンプトや素材が薄いと、訓練データの中央値=「どこかで読んだことのある記事」を出力する。解像度不足も、余計な情報の混入も、この「平均への引き戻し」の現れです。

なので対策は「具体性を入力側で増やす」と「ソース・スコープを締める」の2方向になります。

解像度を上げる

• 素材を自分から渡す。 経験・数値・実際のエラーや判断理由・固有名詞をぶつける。自分の知識ベースの該当ノートを @ で渡すのが一番効きます。Claudeに探させるより、あなたが持っている一次情報を流し込む方が解像度は段違いに上がる。
• 書く前にインタビューさせる。 「執筆前に、解像度を上げるための質問を5個して」と頼む。足りない具体性が炙り出され、平均で埋めるしかなかった穴が埋まります。
• 制約を明示する。 「一般論禁止。各主張に具体例・数値・固有名詞・実コードのいずれかを伴うこと」。これだけで中央値出力をかなり蹴り出せます。
• 切り口（主張）を先に固定する。 「この記事は何を言いたいのか」が無いと、Claudeは無難な総覧記事に流れます。
• 良い段落の見本を1つ渡す（few-shot）。「この解像度で」と示すのが言葉で説明するより速い。

余計な情報を防ぐ

• ソースを固定する。 他記事の混入元はたいてい (a) Web検索 か (b) 訓練データ由来の定番節（「はじめに」「まとめ」的な誰でも書く部分）。(a) は検索をオフにするか「指定したソースのみ参照」と縛る。(b) はアウトライン指定で塞ぐ。
• スコープを宣言させる。 「この記事は X だけ扱う。Y・Z には触れない」を冒頭で固定。
• リサーチと執筆を分離する。 これが一番効きます。まずリサーチ結果をノートファイルに出させ、あなたが剪定してから執筆に進む。本文に入る前に余計な情報を切る検問所を作る。
• アウトラインを先に承認する。 見出し構成を出させ、不要な節をこの段階で削ってから書かせれば、横道に逸れようがない。

Claude Codeでの仕組み化

毎回プロンプトで指示するのは面倒なので、CLAUDE.md かプロジェクトルールに執筆原則を固定するのが現実的です（「一般論を避け具体例必須」「指定ソース以外参照しない」「アウトライン承認前に本文を書かない」など）。さらに記事執筆を「インタビュー → アウトライン承認 → ドラフト」のパイプラインとしてスラッシュコマンドやスキルに落とすと、毎回同じ品質で再現できます。

以前インタビュー型ツールを試していましたが、まさにそのアプローチが解像度問題の核です。「あなたの頭の中にある具体を引き出す工程」を執筆の前段に必ず挟むことが、両方の問題への共通解になります。
