# テストの目的

- 仕様を確定させる
- 仕様を明文化する
- 変更で仕様が意図せず変わったことを検知する
- テストしやすい構造を強制する

仕様を確かめる。

実装を助ける。

利点として、何度も実行できるし、結果もすぐに得られる。

実装を元にテストを書いてもその時点では意味がないけど、とりあえず実装通りにテストを書いておいて、仕様通りにテストを修正して、実装を修正するのには役に立つ。

実装が仕様通りに動いているかどうかは、仕様に対してテストを書かないとわからない。
実装を元にテストを書いても、実装が実装通りに動いているかどうかしかわからない。

確定していない仕様をテストしても、いずれテスト自体が陳腐化する可能性が高い。
まずは試作を繰り返すことで仕様を確定させる。この段階でテストはいらない。手動で都度確認するだけでいい。
仕様が固まってきたらテストで保護した製品版を作る。

何をテストするかは、上位で決めるもの。

- どこまでが「仕様として固めたい」ラインか
- このプロジェクトでStripeのモックを用意するコストは払えるか
- Supabaseはインメモリで差し替えるか、実DBを使うか

このような判断はコードの外にある意思決定。
テストのベストプラクティスに沿っていたとしても、自分たちの方針に合ってないと困る。
