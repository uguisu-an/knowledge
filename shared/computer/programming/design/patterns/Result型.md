Result型、あるいはEither型。
失敗する可能性がある処理の戻り値を扱う型。
RustやHaskellで採用された型の使い方が源流。エラー処理の代わりに、結果の成功・失敗を型として表現する。

明示的なエラー表現のデザインパターン。

成功の場合:

```
{
  ok: true,
  value: T,
}
```

失敗の場合:

```
{
  ok: false,
  error: {
    type: "not_found",
    resource: "Repository",
  }
}
```

このように成功と失敗でそれぞれ違う情報を返す型を定義して、失敗する可能性がある処理の戻り値に使う。`throw`で扱うような例外処理は、予期しない失敗を主に扱う。

Result型は失敗と例外を明確に区別する方法でもある。
失敗はビジネスジックの一部として扱うべきで、予測不能な異常系はビジネスロジックとして定義されていない部分を扱う。
「この処理は失敗しうるが、その失敗は想定内である」という意思表示になる。呼び出し側で対処できない処理を`throw`で扱う。

ページネーションみたいに「途中までの結果」を扱う場合に使う「結果」型はEitherではない。普通に、incompleteでもそうでなくても同じ型を使う。
例えば以下のようなデータ。

```
{
  total_count: 30,
  incomplete_results: true,
  items: [
    // ...
  ],
}
```

Result型はEitherの機能、つまり、okの結果によって型を切り替えるようなデザインパターンが主。
