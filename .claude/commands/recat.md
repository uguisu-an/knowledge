# フォルダ区分の再提案

指定フォルダの **サブフォルダの分け方（区分）そのもの** を批判的に見直し、より良い区分を
再提案する。個々のファイルをどこに置くかではなく、「どんな箱を用意すべきか」を考える作業。
分類規則・ノートの書き方・整理方針は @CLAUDE.md と @docs/organization.md に従う。

ファイルの再配置は **承認後の任意ステップ**。まずは区分案の提示までで一度止まる。

## 原則（この作業の前提）

- **建前ではなく中身を見る**。フォルダ名やその定義（「発達させる」等の抽象動詞）から考えず、
  各ノートの「実際に書かれている主題」からまとまりを作り直す。
- **区分を増やす方向に偏らない**。「この区分は要らない／隣へ吸収できる／development の外
  （`shared/` 直下の `business`・`learning`・`security` など）へ出す」も常に選択肢に入れる。
  キャッチオール（迷ったら入る受け皿）になっている区分は特に疑う。
- **フラット優先・0〜1個のフォルダは作らない・固有名詞は概念と分ける**（organization.md）。

## 区分を診断する4つのものさし

各既存フォルダを、次の観点で点検する:

1. **凝集性** — 中身が単一の主題か。複数の別テーマの寄せ集めなら分割の候補。
2. **排他性** — 隣接フォルダや `shared/` 直下の別フォルダと主題が重複していないか。
   重複するなら境界を引き直すか、片方へ寄せる。
3. **定義の強さ** — 抽象動詞だけで「何でも吸い込める」定義になっていないか。
   良い定義は **隣を排除する**（「Xを扱う。ただしYはこちらでなく◯◯へ」）。
4. **存在意義** — そもそもその区分は要るか。他へ吸収して消せないか。

## 手順

1. **対象を受け取る**。`$ARGUMENTS` のフォルダパスを対象にする。空なら指定を促す。
2. **作業ブランチを切る**。`recat/<対象名>-<YYYY-MM-DD>` のような作業ブランチを作る
   （既に作業ブランチ上ならその旨を伝えて確認する）。
3. **棚卸し**。対象フォルダ直下の全ノートを列挙し、各ノートの **実際の主題を1行** で要約する
   （フォルダ名の建前ではなく中身ベース）。既存サブフォルダの区分と各定義
   （`docs/folder-index.md` や各 README）も突き合わせる。中身が変更されていない
   純粋移動かどうかは `git log` で裏取りしてよい。
4. **現区分を診断**。各既存サブフォルダを上記4つのものさしで判定し、
   「健全／分割すべき／吸収・廃止すべき／development 外へ出すべき」を仕分ける。
5. **ボトムアップでクラスタリング**。建前の区分を一旦外し、ノートの主題で自然なまとまりを作る。
   このとき、対象フォルダの外（`shared/` 直下の別概念）へ出すべきものも候補に挙げる。
6. **新しい区分案を作る**。各候補フォルダに次を付ける:
   - 名前（英語、フラット優先）
   - **隣を排除する1行定義**（「Xを扱う。ただしYは◯◯へ」の形で境界を言語化）
   - そこへ入るノートの一覧
     廃止する区分・新設する区分・対象フォルダ外へ出す区分を、それぞれ理由つきで明示する。
7. **提示して承認を待つ**。次の2点を出す。**承認前に一切書き込まない。**
   - **区分の Before → After**（廃止・新設・外出しと、各新フォルダの1行定義）
   - **ノート単位のマッピング表**（ノート名／現在地／提案先／理由）
8. **（任意）承認後に再配置する**。ユーザーが「再配置まで進めて」と指示した場合のみ:
   - 移動は `git mv`、削除は `git rm`（`mv`/`rm` は使わない）。
   - 空になったフォルダを片付ける。
   - 新フォルダ直下に README を用意する（`概要文 → ## Keywords → ## References` の構成）。
   - `docs/folder-index.md` を更新する。トップレベル構成や概念ツリーに影響するなら
     `docs/structure.md` も更新する。
   - **ノートの中身は断りなく書き換え・統合・削除しない**（移動と配置のみ）。
   - README の `## References` は追加がデフォルト。削除は手順7の提示に明示し承認を得る。
9. **境界の決定を記録する**。今回引いた「隣を排除する1行定義」のうち、今後も迷いそうな
   ものは `docs/organization.md` への追記候補としてユーザーに提案する（次回以降の再現性のため）。
10. **指示があるまでコミットしない**。コミット指示が出たら、リポジトリの慣習に合わせた
    簡潔な日本語メッセージでコミットする（末尾に Co-Authored-By 行）。
