# 概念ツリーの再構成

`docs/structure.md` の概念ツリーと `docs/folder-index.md` を、**各フォルダの README を情報源に**
ボトムアップで再構成し、再提案する。鳥瞰の道具。個別フォルダの中の区分は `/recat` の担当。
整理方針は @CLAUDE.md と @docs/organization.md に従う。

## この作業の前提

- **情報源は README だけ**（主に各 README の 概要・`## Keywords`・`## References`。`## Map` は
  あれば濃い素材だが少数）。9,000 個のノート本体は読まない。README が各フォルダの「定義」で
  ある以上、概念ツリーはその集約であるべき。これにより範囲を有界に保つ。
- **概念ツリーは README の派生ビュー**。両者が食い違うときは **README 側を正** とみなして
  ツリーを直す。ただし README 自体が薄い／ズレている場合は、直すべきフォルダとして指摘し、
  `/recat <folder>` の対象候補に挙げる（ツリーをその場で取り繕わない）。
- **範囲が広いので必ず2段階**。①棚卸し（診断）で一度止まり、②再構成提案へ進む。
  一気に書き換えない。
- **親子はフォルダのネストでなく `## References` で表す**。「A は B の一部」という親子は、
  フォルダを物理的に入れ子にせず A の README が B を References することで示す（flat 維持）。
  実例＝`shared/talk` は `conversation` の子だが、トップレベルの兄弟フォルダのまま
  References で親子を表現している。
- **`## References` は「親」だけでなく「関連」も混在しうる**。実データは方向が一定でない
  （例：`conversation` の References に親 `Communication` と子 `Talk` が同居し、
  `talk`↔`conversation` は相互参照＝循環）。retree は References を鵜呑みにせず、
  **親子（含意）と兄弟（関連）を仕分け**、**循環参照は手順3で検出して報告**する。
- **README の `## References` は追加がデフォルト**。削除は手順5の Before → After に必ず明示し、
  承認を得てから行う（黙って消さない）。
- **`## References` に `shared/` 直下に存在しないフォルダ名が出てきても問題ない**。References は
  トップレベルフォルダへの参照に限定されず、概念的なラベルやサブパス（例：`Well-being` →
  `human/being/well`、`Software Development` → `computer/development`）を指す場合がある。
  棚卸しで「参照先が存在しない」として一律に報告しなくてよい。
- フラット優先。概念ツリーは「どう関連づけて考えているか」の参考であって、フォルダ階層と
  一対一ではない（structure.md の但し書き）。ツリーを精緻にしすぎて階層を深くしないこと。
- **複合概念（複数ドメインにまたがるもの）は「主たる発生文脈」に一箇所だけ置く**。
  例：diet（食事）は Food と Health の複合だが、「何を食べるか」という発生文脈で Food に置き、
  Health との関係は diet の README の `## References` で表現する。
  「どちらの親から調べても意味が通る」とき、最初に概念が発生するドメインを判断軸にする。

## 手順

1. **範囲を受け取る**。`$ARGUMENTS` があればそのサブツリーに限定、無ければ `shared/` 全体。
2. **作業ブランチを切る**。`retree/<YYYY-MM-DD>` のような作業ブランチを作る
   （既に作業ブランチ上ならその旨を伝えて確認する）。
3. **棚卸し（フェーズ1）**。範囲内の各フォルダについて README を収集し、次を**報告して一度止まる**:
   - 各フォルダの 概要 と `## Keywords`／（あれば）`## Map` の要旨（1行）
   - **README が無いフォルダ**（＝ツリーの欠損候補）
   - **概要・Keywords が薄い README**（＝集約の素材にしづらい）
   - **フォーマットの不統一**：`## Keywords` / `## References` / `## Map` の混在、見出しの
     **誤ラベル**（中身と見出しの不一致。例：References 枠に Keywords が入っている）、
     **表記ゆれ**（`## Reference` 単複など）
   - **References の循環・方向不整合**（相互参照や、親と子が同居している References）
     ここで「欠損を先に埋めるか／現状の素材のまま再構成へ進むか」をユーザーに確認する。
4. **ボトムアップ集約（フェーズ2）**。各 README の主題を束ね、自然な上位カテゴリを作る。
   既存ツリーの `Being / Doing / Change …` といった枠は **建前として一旦外し**、README の
   中身から組み直す。既存の枠を残すかどうかも、中身に照らして判断する。
   実フォルダにない中間概念（Nature, Food, Mind, Visual など）を束ね役としてツリーに加えて
   よい。
5. **提示して承認を待つ**。次を出す。**承認前に一切書き込まない。**
   - 概念ツリーの **Before → After**（新設・統廃合・移動した枝と、その理由）
   - `folder-index.md` との整合（定義が変わるフォルダ、索引から漏れているフォルダ）
   - **README 側を直すべきと判明したフォルダの一覧**（`/recat` 対象候補）
6. **（任意）承認後に更新する**。ユーザーが指示した場合のみ:
   - `docs/structure.md` の概念ツリーを書き換える。フォーマット規則:
     全ラベルを CamelCase で統一（実フォルダ名との対応は folder-index.md に委ねる）。
     各項目は独立した箇条書きにし、スラッシュ区切り（`A / B / C`）は使わない。
   - `docs/folder-index.md` を実在フォルダと一致させる。
   - トップレベル構成（ルート `README.md` や `CLAUDE.md`）に波及するなら、その旨を指摘する
     （断りなく書き換えない）。
   - **フォルダの移動・新設・統廃合そのものは行わない**。それは `/recat` の承認フロー経由で行う。
     ここで扱うのは「ツリーと索引というドキュメント」の更新まで。
7. **原則レベルの発見を還元する**。再構成の過程で見えた一般則は `docs/organization.md` への
   追記候補としてユーザーに提案する（次回以降の再現性のため）。
   また、**知識不足で現時点では判断できない欠けている概念**（ツリーに入れるべきかもしれないが
   まだノートが少ない領域など）は `docs/TODO.md` に記録する候補として提案する。
8. **指示があるまでコミットしない**。コミット指示が出たら、リポジトリの慣習に合わせた
   簡潔な日本語メッセージでコミットする（末尾に Co-Authored-By 行）。
