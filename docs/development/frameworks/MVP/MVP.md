# Minimum Viable Product: MVP

「最小限の価値を届けられる製品」の意。

完成しないと品質を高められないので、品質を高めるためにはできるだけ早く完成させないといけない。

できるだけ早く完成させるためには、規模を小さくするのが最善手。

小さく産んで、大きく育てる。

MVP はモックを作って紙芝居でもいい。

[MVP（Minimum Viable Product）とは？実践するメリットと検証方法 | UX MILK](https://uxmilk.jp/65654)

開発するだけで疲弊して、初めから年老いてしまってるサービスもよくある。

MVP は価値を確かめるのに最低限必要な要求なので、単に「必須項目」と言ってもいいかも。できるだけ早く検証を始めるために、必須項目は絞らなければならない。

MVP を作って世に出すのは、コンセプトを明らかにする、その製品の本質を明らかにする意味もある。

車を作るのに、自転車から作るのとは違う。最小限の「車」としてのコンセプトを定義して、検証できるようにする。
車であれば「密閉された空間で（雨風に晒されずに）速く移動できる」もの、とか。

大きな前提から検証する。
https://twitter.com/tsuru___chan/status/1643077131872907270

MVPを作る上で、編集は真っ先に削れるところ。
追加と削除でできるなら必須ではない。

MVPではクライアント側のバリデーションやエラーメッセージの装飾など、あったら便利な機能は省いていい。サーバーで落とせば十分。不便でも問題を解決するのが優先。
オプション機能には何があるか。
クライアントバリデーション、エラー表示、意匠、入力補助、とか。

最初の1つか2つのスプリントでとりあえず触れるもの、小さな問題を解決できるものを作るのが大事。

アジャイル開発で「できることからやる」というのは、大きな製品の部品を作り始めることではない。
小さな問題を解決する小さな製品を作ること。

アジャイル開発では最初から製品を作る。部品ではなく。

ユーザーが積極的に動かしたくなるMVPを作らなければならない。
ユーザーに価値が伝わるMVPは使いたくなる。自分たちのためになることがわかる。

MVPは「実用的に使える」ものでなければならない。試作品・プロトタイプではなく、製品・プロダクトでなければならない。

ユーザーに実際に使ってもらって、フィードバックを得るためのもの。
