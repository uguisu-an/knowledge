# 整理の進め方（retree / recat）

ノートとフォルダの整理は、抽象度の違う3つの層で分担する。**方針 → 鳥瞰 → 深掘り**。
スキルは `.claude/commands/` にあり、`/retree`・`/recat` で呼ぶ。

## 3つの層

| 層     | 道具                   | 役割                                            | 主な成果物                        |
| ------ | ---------------------- | ----------------------------------------------- | --------------------------------- |
| 方針   | `docs/organization.md` | 整理の一般原則（手で育てる）                    | docs のドキュメント群             |
| 鳥瞰   | `/retree`              | 全フォルダの README から概念ツリーを再構成      | `structure.md`・`folder-index.md` |
| 深掘り | `/recat`               | 1フォルダの区分（サブフォルダの分け方）を見直す | そのフォルダの構成・README        |

- **organization.md は原則の置き場**。retree/recat が作業中に見つけた一般則はここへ追記して、
  次回の再現性を上げる（各スキルの最終手順がこの還元を促す）。
- **retree は README だけを情報源**にする。9,000 ノート本体は読まない。README が各フォルダの
  「定義」である以上、ツリーはその集約。範囲を有界に保つための制約。
- **recat はノートの中身**を見る。フォルダ名の建前ではなく、実際に書かれている主題で
  まとまりを作り直す。

## 役割分担（重ならないように）

- **概念の親子・上位カテゴリ**（「service は exchange の一部」等）＝ **retree**。
  ツリーと README の `## References` で表現する。**ノートやフォルダは動かさない**。
- **箱の中身・区分の切り直し・ノートの再配置** ＝ **recat**。実際に `git mv` で動かすのは
  recat の承認フローだけ。
- retree が「README が薄い／ズレている」と判定したフォルダは、その場で取り繕わず
  **recat 候補**として挙げる。

## 使う順番

1. **retree（鳥瞰）を先に**。全体のツリーを再構成し、同時に recat 候補を洗い出す。
2. **recat（深掘り）を後に**。retree が挙げた候補を1フォルダずつ切り直す。

逆順（個別フォルダを全体像なしに切る）は、隣との重複や外出し先を見落としやすい。
retree は薄い README にフラグを立てて先へ進む設計なので、掃除を待たずに回せる。

## 共有する原則

- **親子はフォルダのネストでなく `## References` で表す**（フラット維持）。詳細は
  [organization.md](organization.md)。例：`talk` は `conversation` の子だが、トップレベルの
  兄弟フォルダのまま References で親子を表現している。
- **`## References` は親と関連が混在しうる**。方向が一定でなく循環もある
  （`talk`↔`conversation`）。ツリーを組むときは親子（含意）と兄弟（関連）を仕分け、
  循環は検出して報告する。
- どちらのスキルも **提示で一度止まり、承認前に書き込まない**。`## References` の削除は
  追加と違い、必ず Before → After に明示して承認を得る。
