# Program

日本語では「実行計画」の意。

- プログラムは「計画」である
- プログラムは「実行」できなければならない

処理の手順。

スクリプトをビルドして実行可能な状態になったものもプログラムと呼ぶ。

プログラムは壊れやすい。

program は executive に関わる。計画するだけじゃなくて、「実行される」のが前提。実行できなければ program ではない。計画の中でも特に具体的な部分。

ビジネスの文脈ではプロジェクトを複数持ったものをプログラムという。
プログラムに含まれるプロジェクトの中には、失敗するものもありうる。プログラムとしての大きな目標を達成するのが大事。リカバリーも含めてエグゼクティブの仕事。

- 何を命令するか
- 誰に命令するか

プログラムを実行させる場合、その実行系に理解できる内容で命令しなければならない。

実行系は、うまく処理できる条件が揃っていなければエラーを返す（べき）。
例えば、壁や扉にぶつかって進めないとか。

実行系の限定された命令セットに合わせてプログラムを記述する。

プログラムの実行と記述の問題は分ける。
やりたいことが同じでも記述は違うことがある。

ソフトウェアに限らず、「プログラミング」は実行計画作り。

計画作りでは、着手できる条件を前から揃えられる順序で並べる。

完了の定義: 条件 Z を満たす

1. 作業 A (): 条件 X を満たす
2. 作業 B (条件 X): 条件 Y を満たす
3. 作業 C (条件 Y): 条件 Z を満たす

というように。
このような依存関係になってない作業は分けられる。
依存しない作業を一連の流れに含める必要はない。

逆に、依存関係になっている作業はワンセットで扱わないといけない。

一本道の実行計画になるとは限らない。
順次、分岐、反復の構造で捉える。

ProgramはProjectよりも前にある。

動くソフトウェアの基礎になるのがプログラム。
プログラミングを知らないと動かないソフトウェアを作ってしまう。

「実行」計画である以上、実行可能性は欠かせない。

「誰でも」実行できる、という基準はやりすぎかも。
重要なのは狙った人には実行できるということ。部下であったり、コンピュータであったり。
実行する人に合わせて作らないと実行できない。
