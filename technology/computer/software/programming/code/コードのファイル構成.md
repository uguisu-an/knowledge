フォルダはコンテキストに合わせて区切る。
ドメインとか。

レイヤードアーキテクチャに合わせてコンテキストの中を最初にざっくり区切ることもある。

- `domain`
- `application`
- `presentation`
- `infrastructure`

アプリケーション全体で使うユーティリティはlib, shared, utils, commonといった名前のフォルダに置くことが多い。
最近はsharedをよく使う印象。あるいは単にソースフォルダの直下に置く。

Atomic Designをフォルダにするように、ファイルの粒度でフォルダを分けるべきではない。

Atomic Design や Clean アーキテクチャは抽象概念としては優れているけど、そのままファイルやフォルダを分けるべきではない。
あくまでも概念としてコンポーネントや構造を捉えるために使う。

フォルダは常に文脈で分けるべき。
用途、責務、意図、などを考慮して分ける。とりあえず共通のsharedと、メインのドメイン一つから始める。ブログだったらblogとか。それかsrc/blogと、srcの直下をshared扱いにする。

libに置くのは、アプリケーションの中の共通部品というよりも、再利用可能なモジュール。
sharedはlibに置くような部品も含む。
