# ソフトウェア製品の発達段階

以下の順序で価値を高める。

1. 動かない
2. 動く（最低限問題を解決できる、セキュリティなどの機能も含む）
3. 長く動く（メンテナンスしやすい）
4. 速く動く（パフォーマンスがいい）

最初は設計とか考えるレベルにない。
まずは動くようにして、テストすること。できればテストを自動化すること。

3D ゲームのような重い処理は「動く」のハードルが高い。

銀行や政府系の厳密なシステムもセキュリティや計算精度などで「動く」のハードルが高い。

アプリケーションは、「動く」のが最優先で、次に「長く動く」こと。最後に「より速く動く」こと。
長く動けばそれだけ長く価値を生み出せるし、効率は後から考えればいい。
ただし、「動く」というのはあくまでも有効時間で解決できるという意味で、「より速く動く」のはそれ以上に効率を高めること。
ゲームや AI などの特殊な分野では「動く」のハードルがそもそも高い。
